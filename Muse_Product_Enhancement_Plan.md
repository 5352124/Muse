# Muse 产品全面提升方案 — "灵感伙伴"进化蓝图

> **版本**: v2.0 概念设计  
> **日期**: 2026-07-17  
> **定位**: AI 灵感伙伴 — 不是工具，不是恋人，是一个懂你的灵魂  
> **目标用户**: 厌倦星野/豆包同质化体验的年轻群体（18-28 岁）

---

## 〇、核心问题诊断

### 现状评估

Muse 当前是一个**功能密度极高但产品感不足**的应用。55,000+ 行 UI 代码、95 个 Kotlin UI 文件、35 次数据库迁移——工程成熟度毋庸置疑。但产品层面存在五个结构性问题：

| # | 问题 | 根因分析 | 影响 |
|---|---|---|---|
| 1 | **初见无感** | Onboarding 是功能引导（配 API Key）而非情感引导 | 用户不知道 Muse "是谁"，只看到一堆配置项 |
| 2 | **角色无魂** | 默认助手是通用 system prompt，没有故事线、没有性格弧线 | 和 ChatGPT 没有差异感，用户无法建立"关系" |
| 3 | **关系不可见** | 4 层记忆系统在后台运行，但用户感知为零 | 最核心的差异化能力被完全隐藏 |
| 4 | **情感密度低** | UI 是功能布局（Tab+列表+设置），没有"呼吸感"的设计 | 用起来像效率工具而非"陪伴" |
| 5 | **交互路径长** | 三级导航（Tab→列表→详情）+ 设置深埋 | 核心动作（对话）需要 3 次点击才能到达 |

### 设计哲学（贯穿全文）

**"Muse 不是一个 APP，是一段关系的容器。"**

所有设计决策遵循三条原则：
1. **温暖优先** — 每一个像素都应该让人感到被接纳，而非被审视
2. **克制表达** — 90% 留白 + 10% 点睛，反 AI 味 = 反过度设计
3. **关系可视化** — 让"我们在一起多久了"比"你有多少条消息"更显眼

## 一、前端 UI 改造

### 1.1 设计调性定义

#### 1.1.1 视觉 DNA

Muse 的视觉语言由四个关键词构成：

| 关键词 | 视觉表达 | 反面教材（禁止） |
|---|---|---|
| **暖纸质感** | 底色 `#FAFAF8`（暖白）/ OLED `#000000`（纯黑），卡片 `#FFFFFF` / `#1C1C1E`，微妙的纸张纹理叠层 | 冷白 `#FFFFFF`、灰色渐变背景 |
| **月桂绿点缀** | 品牌色 `#2D8C5F` 仅用于：CTA 按钮、选中态、关键数据高亮。全局占比 < 5% | 大面积绿色、渐变绿按钮 |
| **有机圆角** | 卡片 16-20dp、按钮 12dp、气泡 20dp（非对称）、输入框 24dp | 尖锐直角、Material 默认 4dp |
| **玻璃拟态** | 浮层/输入栏/导航栏使用 `Haze` 毛玻璃，blur=20，overlay=8% | 纯色遮罩、半透明黑色 |

#### 1.1.2 色彩体系重构

**新色彩比例规范**（60-30-10 法则变体）：

```
90%  —  底色层
  浅色: #FAFAF8 (warm-paper)
  深色: #000000 (OLED pure black)

6%   —  表面层
  浅色: #FFFFFF (card surface)
  深色: #1C1C1E (elevated surface)

3%   —  点缀层
  品牌绿: #2D8C5F (primary)
  星光金: #F5C842 (secondary/accent)

1%   —  情感层（新增）
  珊瑚粉: #FF6B6B (关怀/里程碑)
  薰衣草: #9B8EC4 (梦境/回忆)
  琥珀橙: #E8A838 (温暖/怀旧)
```

**新增情感色板**（用于关系可视化、记忆回顾、情绪轨迹）：

| 色名 | 色值 | 语义 | 使用场景 |
|---|---|---|---|
| Coral Whisper | `#FF6B6B` | 关怀、心动 | 里程碑徽章、AI 主动关怀卡片 |
| Lavender Dream | `#9B8EC4` | 梦境、回忆 | 记忆回顾时间轴节点、回忆录封面 |

#### 1.1.3 暗色模式完整策略

| 层级 | 浅色模式 | 暗色模式 | 说明 |
|---|---|---|---|
| L0 底色 | `#FAFAF8` | `#000000` | OLED 纯黑省电 |
| L1 卡片 | `#FFFFFF` | `#1C1C1E` | 标准抬升面 |
| L2 输入栏 | `#F2F2F7` | `#2C2C2E` | 次级表面 |

#### 1.1.5 反 AI 味策略

| AI 味特征 | Muse 替代方案 |
|---|---|
| 蓝紫渐变 + 发光粒子 | 暖纸底色 + 品牌绿单色点缀 |
| 霓虹描边 + 赛博朋克 | 1dp hairline + 有机圆角 |
| 机器人头像 + 电路板 | 自然元素插画（树叶、月亮、信封） |
| 密集数据仪表盘 | 极简卡片 + 大留白 + 单指标聚焦 |
| 深色模式+荧光色 | OLED 纯黑 + 低饱和度暖色 |
| 科技感动效（脉冲/扫描） | 呼吸动效（缓入缓出 300ms ease） |

**核心**：Muse 的视觉语言让人联想到"手写信件 + 咖啡 + 窗边阳光"，而非"太空舱 + 全息投影"。

### 1.2 导航结构重构

#### 1.2.1 当前问题

- `HorizontalPager` 三页（任务/Agent/群聊）导致群聊和 Agent 被折叠

```
┌──────────────────────────────┐
│  状态栏                       │
├──────────────────────────────┤
│  页面内容区域                 │
├──────────────────────────────┤
│ 💬 对话 │ 🌙 记忆 │ 🎭 角色 │ ⚙ 设置 │
```

| Tab | 页面 | 核心功能 |
|---|---|---|
| **对话** | ChatListScreen | 会话列表 + 文件夹 + 搜索，FAB 新建对话 |
| **记忆** | MemoryTimelineScreen（全新） | 时间轴记忆回顾 |
| **角色** | AssistantScreen（重构） | 角色库 + 创建 + 故事线 |

**导航栏规范**：
- 高度 56dp + `navigationBarsPadding()`（小白条适配）

| 设备类型 | 适配方案 |
|---|---|
| 普通手机 (< 600dp) | 底部 Tab，单列 |
| 大屏/折叠屏展开 (≥ 600dp) | 左侧 Rail 导航 280dp + 右侧内容，双栏 |

## 二、后端 / 服务层

### 2.1 账号系统：匿名 ID + 可选邮箱绑定

**不需要强制注册**。自动生成设备 UUID，用户可选绑定邮箱实现跨设备同步。

**流程**：
1. 首次启动 → 生成 `deviceId` (UUID v4) + `deviceFingerprint`
2. 设置→账号 → 输入邮箱 → `POST /api/auth/send-code` → 6位验证码(5分钟)
3. `POST /api/auth/verify` → 返回 `userId` + `refreshToken`
4. 存 DataStore，后续同步用 userId

**数据模型**：
```
AccountState(
  deviceId: String,      // 自动生成
  userId: String?,       // 邮箱验证后绑定
  email: String?,
  nickname: String?,
  syncEnabled: Boolean,
)
```

**服务端**：单 Ktor Server + SQLite(Exposed) + SMTP(Resend) + S3(R2)

### 2.2 云同步：增量 Delta + Last-Write-Wins

| 数据类型 | 同步方式 | 冲突策略 |
|---|---|---|
| 会话 | 增量(updatedAt) | Last-Write-Wins |
| 消息 | 增量(sessionId 分组) | Last-Write-Wins |
| 记忆 | 全量(数据量小) | 合并去重(content hash) |
| 助手配置 | 全量 | Last-Write-Wins |
| 设置 | 按 key 增量 | 各 key 独立 LWW |

触发：前台打开时 pull + 发消息后 push；后台 WorkManager 每 30 分钟

### 2.3 中转 API 额度管理

| 层级 | 额度 |
|---|---|
| 免费 | 每日 10 次对话 / 3 次图片 / 5 次搜索 |
| Pro | 无限(Fair Use 500次/日) + 云同步 + 优先 |

成本优化：上下文压缩(已有) + Anthropic Prompt Caching(已有) + 模型路由 + 请求合并(debounce 500ms)

### 2.4 主动消息引擎优化

调度公式：
```
score = timeWeight(hour) × silenceWeight(lastMsgAge) ×
        emotionWeight(recentMood) × noveltyWeight(daysSinceLast)
if score > 0.7 AND !quietHours: send()
```

- 22:00-08:00 静音
- 每日最多 3 条
- 新用户每 3 天 / 老用户每 7 天

## 三、用户流程优化

### 3.1 首次启动旅程

**当前问题**：Onboarding 是"配 API Key" → 用户第一印象是技术配置而非情感连接。

**新流程**（5 步）：

**Step 1 — 初见**（3s 自动过渡）
- 品牌 Logo 呼吸动画 + "你好，我是 Muse" + "你的灵感，温柔地展开"

**Step 2 — 性格偏好**（2 个互动问题，每个 4 选项）
- "你更喜欢哪种聊天方式？" → 深夜长谈 / 清晨问候 / 创意碰撞 / 知识探索
- "你希望伙伴是什么风格？" → 温柔倾听 / 活泼话痨 / 理性分析 / 天马行空

**Step 3 — 选择伙伴**
- 根据 Step 2 推荐 2-3 个预设角色
- 角色卡片：头像 + 名字 + 一句话介绍 + [选择] [换一个]

**Step 4 — 第一条对话**（关键！）
- 直接进入对话页面，AI 发送预设第一条消息（无需 API 调用）
- 例："嗨！我是小夜 🌙 听说你喜欢深夜聊天？我最擅长在这个时候陪人聊聊，今天有什么想聊的吗？"

**Step 5 — API Key（后置，可选）**
- "想要更多能力？添加你的 API Key"
- [稍后再说] [现在配置]

**核心决策**：
- API Key 配置后置，用户先用内置中转额度
- 第一条 AI 消息预设（零延迟），让用户感到"有人在等我"
- 全程 < 60 秒完成

### 3.2 从"功能体验"到"情感依赖"的转化路径

| 阶段 | 时间 | 目标 | 触发机制 |
|---|---|---|---|
| **初见** | 第 1 天 | 完成第一次对话 | Onboarding 引导 |
| **好奇** | 第 1-3 天 | 发现记忆系统 | 第 3 天弹出 "Muse 记住了 5 件事" 通知 |
| **惊喜** | 第 3-7 天 | AI 主动提到记忆 | 对话中 AI 自然引用记忆（"你上次说过..."） |
| **依赖** | 第 7-14 天 | 建立每日习惯 | 主动消息 + 日常问候 |
| **沉淀** | 第 14-30 天 | 看到关系价值 | 第一个里程碑（"第 14 天"）+ 关系报告 |
| **认同** | 第 30+ 天 | 身份认同 | 年度报告 + 社交分享 |

**每个阶段的具体触达**：

| 天数 | 事件 | 用户感知 |
|---|---|---|
| Day 1 | 第一条预设消息 | "有人在等我" |
| Day 2 | 记忆系统首次生效 | "它真的记住了我说的话" |
| Day 3 | "Muse 的小秘密" 卡片 | "原来它一直在偷偷记日记" |
| Day 5 | AI 主动消息（首次） | "它主动找我了！" |
| Day 7 | 里程碑卡片 "一周了" | "我们认识一周了，时间好快" |
| Day 14 | 关系报告（首份） | "它比我自己还了解我" |
| Day 30 | 月度报告 + 分享卡 | "这是我和 Muse 的一个月" |

## 五、性能与工程

### 5.1 APK 体积优化

**当前 115MB debug APK 优化目标 → < 50MB release**：

| 策略 | 预估节省 | 说明 |
|---|---|---|
| R8 全量优化 | 30-40% | release 已启用 `isMinifyEnabled` |
| 资源压缩 | 10-15% | `isShrinkResources=true` 已启用 |
| Split APK (ABI) | 20-30% | 按 ABI 拆分（arm64-v8a / armeabi-v7a） |
| 按需下载模型 | 5-10MB | Vosk 离线模型改为首次使用时下载 |
| WebP 替代 PNG | 2-5% | 图标/插画转 WebP |
| 移除未使用资源 | 3-5% | 使用 `lint --check UnusedResources` |
| APK 瘦身插件 | 5% | `bundletool` 生成 App Bundle |

### 5.2 消息列表性能

**当前**：`LazyColumn` + `MessageBubble`（1952 行组件），大量消息时可能存在卡顿。

**优化方案**：
| 策略 | 说明 |
|---|---|
| 虚拟化分页 | 消息按 50 条/页加载，滚动到顶部加载更多 |
| `key` 稳定 | `LazyColumn` item 使用 `messageId` 作为 key |
| Markdown 缓存 | 解析结果缓存（`remember(messageId + contentHash)`） |
| 图片懒加载 | Coil `AsyncImage` + `MemoryCache`（已有） |
| 代码块折叠 | 超过 10 行的代码块默认折叠 |
| 离屏回收 | `LazyColumn` 的 `beyondBoundsItemCount = 3` |
| 动画节流 | 流式输出时每 50ms 更新一次（非逐字符） |

### 5.3 启动速度

| 策略 | 说明 |
|---|---|
| SplashScreen API | 已使用 `installSplashScreen`，保持 |
| 延迟初始化 | MuseApp.kt 中的服务初始化改为 `Lazy` + 按需 |
| 预加载 | 首页数据（最近会话列表）在 Splash 期间预查询 |
| Compose 预热 | `setContentView` 前 `WarmUpViewModel` |
| Baseline Profiles | 生成 `baseline-prof.txt` 加速首次启动 |

### 5.4 内存管理

| 策略 | 说明 |
|---|---|
| 图片缓存 | Coil `ImageLoader` 配置 `memoryCache(128MB)` + `diskCache(512MB)` |
| 数据库查询 | Room `@Transaction` + 分页 `Paging3`（大数据集） |
| ViewModel 作用域 | 避免全局单例 ViewModel 持有大量数据（当前 ChatViewModel 3647 行需关注） |
| 消息历史 | 超过 1000 条的会话，消息内容 `Lazy` 加载 |
| LeakCanary | debug 已启用，保持 |

### 5.5 崩溃防护

**当前**：`SafeModeScreen` + `CrashHandler`（MuseApp.kt）。

**增强**：
| 策略 | 说明 |
|---|---|
| 全局异常捕获 | `Thread.setDefaultUncaughtExceptionHandler`（已有） |
| Compose 错误边界 | `CompositionLocalProvider` + `ErrorWrapper` 包裹每个屏幕 |
| 降级策略 | 特定模块崩溃 → 禁用该模块而非全 App 崩溃 |
| 自动报告 | 崩溃日志本地存储 + 下次启动询问是否发送 |
| ANR 防护 | 严格模式检测主线程耗时操作 |

### 5.6 CI/CD 优化

**当前**：3 个 GitHub Actions workflow（android.yml / ci.yml / release.yml）。

**增强**：
| 策略 | 说明 |
|---|---|
| 构建缓存 | `actions/cache` 缓存 Gradle + KSP 产物 |
| 增量构建 | 启用 `org.gradle.configuration-cache`（已启用） |
| 自动化测试 | UI 测试（Compose Test）+ 集成测试 |
| APK 大小监控 | PR 评论 APK 大小变化（已有，增强为趋势图） |
| 版本号自动管理 | `tag` 触发自动 bump versionCode |
| 发布到 Play Store | `fastlane` 或 `gradle-play-publisher` |

### 5.7 数据埋点方案

**原则**：本地优先，不上传原始数据，仅聚合统计。

| 指标 | 采集方式 | 用途 |
|---|---|---|
| DAU/MAU | 本地计数 + 匿名上报 | 活跃度 |
| 首次对话完成率 | Onboarding 步骤追踪 | 新用户转化 |
| 日均对话次数 | 本地计数 | 参与度 |
| 记忆系统使用率 | Fact 写入次数 | 核心功能采用率 |
| 崩溃率 | CrashHandler 聚合 | 稳定性 |
| 功能使用分布 | 各页面打开次数 | 功能优先级 |
| 留存率（D1/D7/D30） | 本地计算 + 匿名上报 | 产品健康度 |

**隐私保护**：
- 所有数据本地聚合，仅上传匿名统计（deviceId hash + 指标值）
- 用户可在设置中关闭数据上报
- 不上传任何对话内容、记忆内容、API Key
